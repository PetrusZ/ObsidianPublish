    # GPT 是什么

    Generative Pre-trained Transformer： 生成式预训练变换模型

    Generative：自发生成新的文本序列，可以用于自然语言处理任务。

    Pre-trained：在大规模的数据集上进行无监督学习，通过学习特征/知识，增加模型的反话能力和表现。

    Transformer：基于自注意力机制的神经网络结构。可以理解为一个黑盒子，将自然语言翻译为machine learning语言。

    New bing 回答：

    GPT的技术细节比较复杂，我尽量简单地给你讲一下。GPT是一种基于Transformer架构的语言模型，它使用了自注意力机制（self-attention mechanism）来捕捉文本中的长距离依赖关系。GPT的特点是使用了生成式预训练（generative pre-training）和精细调节（fine-tuning）的方法，即先在大规模的无监督语料上进行预训练，学习文本的通用表示，然后在小规模的有监督数据集上进行精细调节，适应不同的下游任务。GPT还使用了掩码（masking）技术，使得模型只能看到文本中当前位置之前的内容，从而实现自回归（autoregressive）生成。

    # GPT 基础原理

    • ChatGPT的实质功能是单字接龙
    • 长文由单字接龙的自回归所生成
    • 通过提前训练才能让它生成人们想要的问答
    • 训练方式是让它按照问答范例来做单字接龙
    • 这样训练是为了让它学会「能举一反三的规律」
    • 缺点是可能混淆记忆，无法直接查看和更新所学，且高度依赖学习材料

    # ChatGPT 的三个训练阶段

    • “开卷有益”阶段：让 ChatGPT 对「海量互联网文本」做单字接龙，以扩充模型的词汇量、语言知识、世界的信息与知识。使 ChatGPT 从“哑巴鹦鸱”变成“脑容量超级大的懂王鹦鹉”。
      • 无监督学习
    • “模板规范”阶段：让 ChatGPT 对「优质对话范例」做单字接龙，以规范回答的对话模式和对话内容。使 ChatGPT 变成“懂规矩的博学鹦鹉”。
      • 监督学习
    • “创意引导”阶段：让 ChatGPT 根据「人类对它生成答案的好坏评分」来调节模型，以引导它生成人类认可的创意回答。使 ChatGPT 变成“既懂规矩又会试探的博学鹦鹅”。
      • 强化学习

    当单字接龙模型的规模达到一定程度后 (LLM），就会涌现出“理解”指令，“理解”例子，思维链的能力。

    # Completions

    Completions endpoint 是 ChatGPT API 的核心，它提供了一个极其灵活和强大的简单接口。你输入一些文字作为promot，API 将返回一个文本completion，它会尝试匹配你提供的任何指令或上下文。

    可以将其视为非常高级的自动补全功能——该模型会处理你的文本提示并尝试预测接下来最有可能出现的内容。

    # 核心概念

    # Prompts

    设计 prompt 本质上就是你如何"编程"模型，通常是提供一些指令或例子。

    # Tokens

    模型通过将文本分解为 tokens 来理解和处理文本。Tokens 可以是单词或只是字符块。例如，“hamburger”可以被分解为 “ham”、“bur” 和 “ger”，而像"pear"这样简短、常见的词是一个单一的 token。

    在给定的 API 请求中处理的 token 数量取决于输入和输出的长度。作为一个粗略的经验法则，一个 token 约等于 4 个字符或 0.75 个英文单词。要记住一个限制是你的文字 prompts 和生成的 completion 组合不能超过模型的最大上下文长度（大多数模型是 2048 个 tokens，大约 1500 个单词）。

    可以通过 tokenizer tool 来学习关于文本是如何转成为 tokens 。

    # Text Completion

    Completion endpoint 可用于各种各样的任务。它为任何 model 提供了一个简单但功能强大的接口。输入一些文本作为 prompt，model 将生成一个文本补全，尝试匹配你提供的任何上下文或模式。

    每次看到的实际补全情况可能有所不同，因为默认情况下 API 是不确定的。这意味着即使你的 prompt 保持不变，每次调用它时可能会得到略微不同的补全。将 temperature 设置为 0 将使输出大部分具有确定性，但仍然可能会保留少量可变性。

    # Chat Completions

    # 介绍

    Chat model 将一系列 messages 作为输入，并返回 model 生成的 message作为输出。

    尽管聊天格式旨在简化多回合对话，但它对于没有任何对话的单回合任务同样有用（例如之前由指令遵循模型（如 text-davinci-003）提供的任务）。

    # Note: you need to be using OpenAI Python v0.27.0 for the code below to work
import openai

openai.ChatCompletion.create(
  model="gpt-3.5-turbo",
  messages=[
        {"role": "system", "content": "You are a helpful assistant."},
        {"role": "user", "content": "Who won the world series in 2020?"},
        {"role": "assistant", "content": "The Los Angeles Dodgers won the World Series in 2020."},
        {"role": "user", "content": "Where was it played?"}
    ]
)

    其主要输入是 messages 参数，messages 必须是一个 message 对象数组，其中每个对象都有一个角色（“系统”、“用户”或“助手”）和内容（消息的内容）。

    通常，对话首先使用 system message 进行格式化，然后是交替的 user 和 assistant messages。

    system message 有助于设置 assistant 的行为。在上面的例子中，assistant被指示“You are a helpful assistant.”

    user message 有助于指导 assistant。它们可以由应用程序的最终用户生成，或由开发人员设置为指令。

    assistant message帮助存储先前的响应。它们也可以由开发人员编写，以帮助提供所需行为的示例。

    当用户指令引用先前的消息时，包括对话历史记录会有所帮助。在上面的示例中，user 的最后一个问题是“Where was it played?？”仅在有关 World Series of 2020的先前消息的上下文中才有意义。由于模型对过去的请求没有记忆，因此必须通过对话提供所有相关信息。如果对话超过模型的token限制，则需要以某种方式缩短它。

    # Instructing Chat Models

    指导 model 的最佳实践可能因 model 版本而异。以下建议适用于 gpt-3.5-turbo-0301，可能不适用于未来的 model。

    许多对话以 system message 开始，以温和地指示 assistant。例如，这是用于 ChatGPT 的 system message之一：

    1

    You are ChatGPT, a large language model trained by OpenAI. Answer as concisely as possible. Knowledge cutoff: {knowledge_cutoff} Current date: {current_date}

    通常，gpt-3.5-turbo-0301 对 system message 的关注度不高，因此重要的指令往往放在 user message中比较好。

    如果 model 没有生成你想要的输出，可以随意迭代并尝试潜在的改进。可以尝试以下方法：

    • 让你的指令更明确
    • 指定你想要答案的格式
    • 在确定答案之前让 model 逐步思考或讨论利弊

    如需 prompt 的复杂的工程创意，可以阅读有关  techniques to improve reliability 的 OpenAI Cookbook Guide。

    除了 system message 之外，temperature 和 max tokens 是开发人员必须影响 chat model 输出的众多选项中的两个。对于 temperature，较高的值（如 0.8）将使输出更加随机，而较低的值（如 0.2）将使输出更加集中和确定。而关于 max tokens，如果要将响应限制为特定长度，可以将 max tokens 设置为任意数字。但这也可能会导致问题，例如，如果将 max tokens设置为 5，输出将被切断并且结果对用户没有意义。

    # Chat VS Completions

    由于 gpt-3.5-turbo 的性能与 text-davinci-003 相似，但每个 token的价格低 10%，因此建议在大多数用例中使用 gpt-3.5-turbo。

    # 使用技巧

    # Prompt Design

    GPT模型可以做任何事情，从生成原创故事到执行复杂的文本分析。因为他们可以做很多事情，所以你必须明确描述你想要什么。通常良好提示的秘诀是展示，而不仅仅是讲述。

    以下是创建prompt的三个基本准则：

    展示和讲述。 通过指令、示例或两者的结合，来明确你需要什么。如果你希望模型按字母顺序对项目列表进行排名或按感情对段落进行分类，请向它展示你想要的，也就是说提供一个例子。

    提供高质量数据。 如果你正在尝试构建一个分类器或让模型遵循某种模式，请确保有足够的示例。并且一定要校对你的例子——模型通常非常聪明，可以看穿基本的拼写错误并给你一个回复，但它也可能认为这是故意的，这会影响回复。

    检查你的设置。 temperature 和 top_p 设置控制模型在生成回复时的确定性。如果你要求它提供只有一个正确答案的答复，则需要将这些设置得较低。如果你正在寻找更多样化的回复，则可能希望将它们设置得更高。在使用这些设置时犯的第一个错误是假设它们是“聪明”或“创造力”控制。

    # 扩展功能

    # ChatGPT API 相比 ChatGPT 的一些优点

    • 显式定义角色。在新的 ChatGPT API 中，消息增加了 role 的属性，表示其所属的角色，其中 system 用于定义接口的行为，比如「你是一个写作助手」， user 和 assistant 则用于区分用户输入和模型输出。这使得创造特定需求的助手变得更加清晰明确
    • 选择性地去掉会话中的信息。ChatGPT 最大可以存储 4096 个 token（大约16384个英文字母）的上下文，当一个会话的内容超出这个数量，最前面的信息就会被遗忘。而 ChatGPT API 每次都要将完整的上下文传递过去，这意味着我们可以选择保留重要的信息，选择性地去掉一些无用的以避免超出限制。
    • 返回多个结果供选择。通过传递 n 参数，可以一次性返回多个不同的结果，适合文字润色、短内容生成等场景，避免多次重复问询。
    • 使用 logit_bias 参数调整特定词汇（token）在结果中出现的可能性，实现减少重复甚至屏蔽特定词汇的功能。
    • 使用 temperature/top_p 参数调整结果的相关性和准确性。当我们需要发散思维、拓宽脑洞时，可适当调低结果的相关性；而当提供的上下文足够多，需要做精确的分析时，则可调高相关性，减少 ChatGPT 编造内容的可能。

    # Fine-tune

    # 介绍

    微调可让你从 API 提供的 model中获得更多收益：

    1. 比 prompt design 更高质量的结果
    2. 能够训练比 prompt 可容纳的更多的例子
    3. 由于更短的 prompt 而节省了 tokens
    4. 更低的延迟请求

    GPT-3 已经在互联网的大量文本上进行了预训练。当给出仅包含几个示例的 prompt 时，它通常可以凭直觉判断出你要执行的任务并生成合理的 completion。这通常称为“小样本学习”。

    而微调可以通过训练，比仅在 prompt 中可容纳的更多例子，来改进小样本学习，让你在大量的任务中取得更好的结果。对模型进行了微调后，你不再需要在 prompt 中提供示例。

    在高层次上，微调涉及以下步骤：

    1. 准备和上传训练数据
    2. 训练新的微调模型
    3. 使用你的微调模型

    微调模型训练和使用的定价可参考 价格页面。

    # 哪些模型可以微调？

    微调目前仅适用于以下基本模型：davinci、curie、babbage 和 ada。这些是训练后没有任何指令的原始模型（例如 text-davinci-003 有经过指令训练）

    # 准备数据

    数据必须是 JSONL 文档，每一行都是一个 prompt-completion 对，对应一个训练示例。

2
3

    {"prompt": "<prompt text>", "completion": "<ideal generated text>"}
{"prompt": "<prompt text>", "completion": "<ideal generated text>"}
{"prompt": "<prompt text>", "completion": "<ideal generated text>"}

    设计微调用的 prompts 和 completions 与使用基础 model 时的设计 prompts 不同。特别是，基础 model 的 prompts通常包含多个示例（“小样本学习”），对于微调，每个训练示例通常由一个输入示例以及相关输出组成，无需提供详细指令或在同一 prompts 中包含多个示例。

    训练示例越多越好。推荐至少有几百个示例。一般来说，发现数据集大小每增加一倍都会导致模型质量线性增加。

    # Embeddings

    # Embeddings 是什么

    Texting embeddings 可衡量文本字符串的相关性。Embeddings 通常用于：

    • 搜索（结果按与查询字符串的相关性排序）
    • 聚类（其中文本字符串按相似性分组）
    • 推荐（推荐具有相关文本字符串的项目）
    • 异常检测（识别出相关性很小的异常值）
    • 多样性测量（分析相似性分布）
    • 分类（其中文本字符串按其最相似的标签分类）

    Embedding 是一个浮点数的 vector（list）。两个 vector 之间的距离衡量它们的相关性。小距离表示高相关性，大距离表示低相关性。

    # FAQ

    如何快速检索 K 个最近的 embedding vectors？

    快速搜索多个 vector，建议使用 vector database。

    Vector database 选项包括：

    • Pinecone，一个完全托管的 vector database
    • Weaviate，一个开源 vector 搜索引擎
    • Redis 作为 vector 数据库
    • Qdrant，vector 搜索引擎
    • Milvus，一个为可扩展相似性搜索而构建的 vector database
    • Chroma，一个开源 embeddings store
    • Typesense，快速开源 vector 搜索
    • Zilliz，数据基础设施，由 Milvus 提供支持

    # Plugins Limited Alpha

    # 介绍

    OpenAI 插件将 ChatGPT 连接到第三方应用程序。这些插件使 ChatGPT 能够与开发人员定义的 API 进行交互，从而增强 ChatGPT 的功能并允许其执行范围广泛的操作。

    • 插件可以允许 ChatGPT 执行以下操作：
    • 检索实时信息；例如，体育比分、股票价格、最新消息等。
    • 检索知识库信息；例如，公司文档、个人笔记等。
    • 代表用户执行操作；例如，订机票、订餐等。

    插件开发者通过标准化的 manifest 文件和 OpenAPI 规范暴露一个或多个 API endpoints。这些定义了插件的功能，允许 ChatGPT 消化文件并调用开发人员定义的 API。

    AI model 充当智能 API 调用者。给定 API 规范和何时使用 API 的自然语言描述，model 会主动调用 API 来执行操作。例如，如果用户问“我应该在巴黎住几晚？”，model 可能会选择调用酒店预订插件 API，接收 API 响应，并结合 API 数据和自然语言能力生成面向用户的答案。

    # Example

    • build a semantic search and retrieval plugin
      • 该插件可以对个人或组织文档进行语义搜索和检索。它允许用户通过提问或用自然语言表达需求，来从他们的数据源（如文件、笔记或电子邮件）中获取最相关的文档片段。企业可以使用此插件通过 ChatGPT 向员工提供内部文档。
      • 该插件使用 OpenAI 的 text-embedding-ada-002  embeddings model 来生成文档块的 embeddings，并使用后端的 vector database 来存储和查询它们。作为一个开源和自托管的解决方案，开发人员可以部署自己的检索插件并将其注册到 ChatGPT。检索插件支持多个 vector database providers，允许开发人员从列表中选择他们喜欢的一个。

    # 短期应用场景

    • codex 补全 yaml
      • 应用场景较少，需要自己写 yaml 的情景不多
    • 工作流 yaml 解释
      • 有一定作用，但会将工作流 yaml 的内容进行较大幅度的精简
    • 生成构建脚本
      • 不理解项目无法生成
    • 构建脚本 lint、解释
    • dockerfile lint、解释
      • 同上
    • 局限
      • 应用场景仅限于片段的 yaml 和脚本
      • 无法理解整个项目
      • 如何理解整个项目
        • 用文字的方式将项目结构展现出来
        • 将所有代码上传，让 ChatGPT 学习理解
    • 贴近开发者日常场景，比如构建过程、部署过程、环境使用，让开发者更加便利的诊断定位问题
      • 需要让 ChatGPT 完全理解 zadig，理解 zadig 的错误信息，目前没有很好的解决方案
      • Fine-tune？仅适用于基础 model不支持 Chat

    # 长期应用场景

    • 核心业务（devops）的小场景，浏览器插件开发，比如工作流 yaml 校验？环境状态巡检？工作流执行？
      • 工作流 yaml 校验：按理说可行，但测试后效果不佳
      • 环境状态巡检：已有开源项目，k8sGPT
      • 工作流执行：Plugin
    • 工程师日常的操作，比如看执行过程中的日志、环境实时日志、任何错误返回后迅速给一些 AI 建议？
      • 日志太长
      • 错误信息需要让 ChatGPT 学习、理解
        • fine-tuning？
    • 基于 azure 服务，提供面向企业内部的代码提示工具（目前 github copilot 只能是 github 用户） 。在 zadig 里提供一个比如插件或者区域可以给开发者做 Prompt
      • yaml ? or code ?
    • 将数据喂给 ChatGPT 统一问答入口为开发者服务。ChatGPT 可以自动地对帖子中的关键词进行识别，并为用户提供一些与其问题相关的文章或已知解决方案，或者可以向用户提出一些澄清性问题，以便更好地理解其问题，并提供相应的答案。
      • 部分可实现
      • 关键字识别能力较弱
    • 给 zadig 提供自然语言界面。
      • Plugin
    • 工作流全流程日志（构建、部署、测试等）、运行时应用实时日志，与 ChatGPT 结合分析诊断问题并进行优化，做质量洞见。
      • 日志太长，有效信息太少
    • 结合 Zadig 产生的研发过程数据进行数据洞察，通过自然语言处理，识别数据中可能被忽略的模式和趋势，并为企业提供更有意义的见解。
      • ChatGPT 没有自己的思维、想法，还是需要用大量数据对其进行训练，才能产生自己的见解。小样本效果待定。
      • Lilian 或 Minmin 简单测试一下
      • Plugin 接入
    • ChatGPT 深度集成 zadig，可以根据 prompt，自动生成构建脚本，创建并执行工作流，实现构建、部署等一系列任务，完成一个完整的 devops 流程。
      • 其中 ChatGPT 实现分解任务、生成脚本、接入平台，分发任务等。
      • 因为深度集成，需要 ChatGPT 插件或者定制开源实现，来完成此功能。
      • ChatGPT tokens 限制（其他模型实现项目分析）
      • 安全风险，将全部代码上传到ChatGPT
